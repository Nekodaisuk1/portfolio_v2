# ポートフォリオサイト 要件定義書

## 1. 文書の目的
本書は、複数のセルを自動配置するポートフォリオサイトの実装要件を定義するものである。実装担当エンジニアが、設計意図・表示ルール・管理機能・非機能要件を誤解なく把握し、保守しやすい構成で開発できることを目的とする。

---

## 2. プロジェクト概要

### 2.1 サイトのコンセプト
- 複数の四角・丸のセルを敷き詰めるように自動配置するポートフォリオサイトを構築する。
- セルには、情報を持つものと装飾用のものが存在する。
- HOME では各ページのセルを混在表示し、各下層ページでは絞り込まれたセルのみを表示する。
- 表示に毎回の変化を持たせるため、ランダム生成される装飾セルを導入する。
- デザイナーが Figma で作成したデザインに後から実装側が合流しやすい構造とする。

### 2.2 実装方針の要点
- レイアウトは **仮想グリッドベースの自動配置** を採用する。
- 丸セルもレイアウト上は矩形占有として扱い、見た目のみ border-radius で表現する。
- HOME と下層ページでセル構築ロジックを分離する。
- セルの位置固定機能は **初期実装では作らない**。
- 作品セル詳細はページ遷移ではなく、重ね表示のウィンドウで表示する。
- 管理ページからセルおよび表示設定を編集可能にする。

## 4. セルの分類

### 4.1 コンテンツセル
情報を持つ主要セル。

#### 例
- ABOUT への導線セル
- WORKS の作品セル
- CONTACT の案内セル
- HOME 上での各ページ遷移セル

#### 主な属性
- id
- title
- description
- image
- link
- targetPage
- visiblePages
- shape
- size
- priority
- tags
- published
- contentKind

### 4.2 TECHセル
ユーザーが明示的に追加する装飾セル。視覚効果や世界観補強が目的。

#### 表示ルール
- HOME では **全 TECH セルを表示** する。
- ABOUT / WORKS / CONTACT では **表示数を絞る**。
- HOME 用と下層ページ用で **共通ロジックを使わない**。
- 絞り方は後に定義可能なよう、設定拡張に耐える構造にする。
- 表示設定は **ADMIN から変更可能** とする。

#### 主な属性
- id
- name
- visualType
- shape
- size
- styleToken
- visiblePages
- priority
- published

#### 表示設定属性案
- homeDisplayMode
- subpageDisplayMode
- subpageDisplayCount
- subpageSelectionRule

### 4.3 デブリセル
ランダムに追加される小型の無情報セル。隙間埋め、見た目の揺らぎ、密度感調整を担う。

#### 表示ルール
- デブリセル数は **コンテンツ数に応じて調整** できること。
- 少なくとも **HOME 以外では控えめ** にすること。
- 生成数は以下いずれかで設定可能とする。
  - 固定値
  - ランダム範囲
  - コンテンツ数比例
- 設定は **ADMIN から変更可能** とする。

#### 特徴
- 情報を持たない
- 主に 1x1 を基本とする
- 訪問ごと、またはセッションごとに生成される
- 表示ページごとの有効化・無効化が可能

#### 主な属性
- id
- shape
- size
- styleToken
- generatedAt
- seed
- visiblePages

#### 生成設定属性案
- generationMode (`fixed` / `range` / `contentRatio`)
- fixedCount
- minCount
- maxCount
- ratioPerContent
- pageWeight

## 5. ページ別表示要件

### 5.1 HOME

#### 表示対象
- ABOUT / WORKS / CONTACT に属するコンテンツセル
- 全 TECH セル
- ランダム生成されたデブリセル

#### 要件
- 初期表示領域に、**ABOUT / WORKS / CONTACT への遷移セルが必ず 1 つ以上含まれること**。
- 表示内容は全ページ混在とする。
- 主要導線セルは埋もれないよう優先配置する。
- デブリセルにより訪問ごとの差分を持たせる。
- HOME 用 TECH 表示ロジックは下層ページと分離する。

#### 作品セルクリック時の挙動
- ページ遷移ではなく、**重ね表示ウィンドウ**を出現させる。
- ウィンドウはブラウザ風の見た目を持つ。
- ウィンドウ内に **WORKS ページへの誘導リンク** を配置する。

### 5.2 ABOUT

#### 表示対象
- ABOUT に属するコンテンツセル
- ABOUT 向けに絞られた TECH セル
- 必要なら ABOUT 用デブリセル

#### 要件
- TECH 表示数は HOME とは別設定で管理する。
- デブリセルは HOME より控えめにする。
- WORKS 専用セル、CONTACT 専用セルは表示しない。

### 5.3 WORKS

#### 表示対象
- WORKS に属するコンテンツセル
- WORKS 向けに絞られた TECH セル
- 必要なら WORKS 用デブリセル

#### 要件
- タグ・カテゴリによる将来拡張に耐えること。
- 作品セルクリック時はページ遷移ではなく、**重ね表示ウィンドウ**を表示する。
- ウィンドウ内に、一覧性を保つ導線または WORKS ページへの導線を持たせる。

### 5.4 CONTACT

#### 表示対象
- CONTACT に属するコンテンツセル
- CONTACT 向けに絞られた TECH セル
- 必要なら CONTACT 用デブリセル

#### 要件
- 外部リンク、メール、SNS 等への導線を持てること。
- TECH 表示数は HOME とは別設定で管理する。
- デブリセルは HOME より控えめにする。

## 6. レイアウト要件

### 6.1 基本方針
- セルは **自動配置** される。
- 配置方式は **仮想グリッドベース** とする。
- 丸セルも占有判定上は矩形として扱う。

### 6.2 配置優先順位
1. HOME の主要導線セル
2. 各ページの主要コンテンツセル
3. 通常コンテンツセル
4. TECH セル
5. デブリセル

### 6.3 サイズ
初期実装では以下のみ対応する。
- 1x1
- 2x1
- 2x2

### 6.4 初期表示保証
HOME では初期ビューポート内に以下が必ず視認できること。
- ABOUT 遷移セル
- WORKS 遷移セル
- CONTACT 遷移セル

### 6.5 位置固定機能
- **現時点では実装しない**。
- 全セルを自動配置対象とする。
- 型や UI に中途半端な固定機能を先に入れない。

---

## 7. ウィンドウ表示要件

### 7.1 対象
- HOME 内の作品セル
- WORKS 内の作品セル

### 7.2 挙動
- 背景ページはそのまま残る。
- その上にブラウザ風ウィンドウを表示する。
- ページ遷移はしない。
- 閉じるボタンで閉じられる。
- MVP では単一ウィンドウのみ扱う。

### 7.3 表示内容
- タイトル
- サムネイルまたはビジュアル
- 概要
- 使用技術
- 外部リンク
- WORKS ページへの誘導リンク

### 7.4 非対象
- ABOUT / CONTACT の通常導線セルは通常遷移でよい。
- すべてのセルをウィンドウ表示対象にはしない。

## 9. データ設計方針

### 9.1 DB 採用方針
- ORM は **Prisma を前提** とする。
- ただし、導入タイミングは UI 骨格と主要ロジック確定後を推奨する。
- 初期に DB 化する対象は以下とする。
  - cells
  - page_settings
  - work_details

### 9.2 モデル方針
- `cells`: content / tech を一元管理
- `page_settings`: ページ別 TECH 表示・デブリ設定管理
- `work_details`: 作品ウィンドウ用詳細データ
- `debris`: DB 保存せず、設定に基づいて動的生成でよい

## 10. 実装構成方針

### 10.1 想定技術構成
- Next.js App Router
- TypeScript
- Tailwind CSS（Figma トークンに合わせた設計）
- Prisma
- React Hook Form
- Zod
- 必要に応じて Zustand（作品ウィンドウ状態のみ）

### 10.2 ディレクトリ責務の基本方針
- `design/`: Figma 由来のトークンやデザインレシピ
- `components/`: UI 描画のみ
- `features/portfolio/`: セル構築ロジック、絞り込み、配置
- `server/` または `repositories/`: DB アクセスとサービス層
- `app/`: ページ単位の構成

### 10.3 責務分離のルール
- UI コンポーネント内で配置ロジックを書かないこと。
- HOME と下層ページのセル構築ロジックを分けること。
- Figma 上の見た目都合を直接 DB 設計に持ち込まないこと。
- 管理画面の都合で公開側の型を汚さないこと。

## 12. 非機能要件

### 12.1 保守性
- 型定義を明示すること。
- マジックナンバーを避けること。
- セル種別判定を enum または定数で管理すること。
- 表示、データ取得、配置計算、管理操作を混在させないこと。

### 12.2 拡張性
以下の将来拡張に耐える構造とする。
- ページ追加
- セル形状追加
- 作品カテゴリ追加
- フィルタ追加
- アニメーション追加
- CMS 接続
- 認証付き管理画面
- TECH 選定ルール高度化
- デブリ生成ルール高度化

### 12.3 レスポンシブ
- PC / Tablet / Mobile に対応すること。
- 列数のみ切り替え、配置アルゴリズム自体は共通化すること。

### 12.4 パフォーマンス
- 初回表示で過剰な再計算をしないこと。
- レイアウト計算はメモ化可能な形にすること。
- デブリ生成と配置計算を分離すること。
- モーダル表示時に全体レイアウトを再計算しないこと。

## 15. エンジニア向け注意事項
- HOME と下層ページのセル構築ロジックは分けること。
- デブリ数算出は設定差し替え可能な純粋関数にすること。
- レイアウトロジックは UI と分離すること。
- 作品ウィンドウは overlay として独立責務にすること。
- 初期段階では DB より UI / ロジックの安定を優先してよい。
- Figma の見た目に追従しやすいよう、デザイン由来の値はトークンとして集約すること。
